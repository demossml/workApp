1. # ИСПОЛНИТЕЛЬНОЕ РЕЗЮМЕ

workApp — это Telegram Mini App для управления сетью розничных магазинов с
интеграцией кассовой системы Evotor. В системе уже накоплена богатая
аналитическая база: данные по выручке, возвратам, сотрудникам, расписаниям,
остаткам товаров. AI-директор — это слой интеллекта поверх этих данных, который
превращает сырые цифры в конкретные управленческие решения.

ЧТО УЖЕ ЕСТЬ В ПРОЕКТЕ:
• POST /api/ai/procurement-recommendations — рекомендации закупки (MVP готов)
• POST /api/ai/employee-shift-kpi — KPI сотрудников (MVP готов)
• GET /api/analytics/revenue/hourly-plan-fact — план-факт по часам
• GET /api/analytics/revenue/refund-documents — документы возвратов
• Cloudflare Workers + D1 + Hono — современный и масштабируемый стек
• Telegram Bot с кнопками и Mini App
• Cloudflare Cron Triggers — настроены, но не используются для AI
• Cloudflare KV — доступен для хранения контекста диалога

ГЛАВНАЯ ПРОБЛЕМА ТЕКУЩЕГО AI:
• AI работает в режиме PULL — директор должен сам спрашивать
• Нет нарративов: цифры без объяснения причин
• Нет проактивных алертов при аномалиях
• MVP endpoints существуют, но не используются в интерфейсе
• Нет расписания автоматических отчётов
• Нет диалогового режима с данными

КЛЮЧЕВЫЕ ПРИНЦИПЫ AI-ДИРЕКТОРА:

1. Push-first, не pull — AI сам сообщает важное, директор не должен
   открывать приложение чтобы узнать о проблеме
2. Данные → Контекст → Действие — каждый инсайт заканчивается конкретной
   рекомендацией «Сделать X», а не «обратите внимание»
3. Роль-зависимые ответы — SUPERADMIN видит все магазины, ADMIN — свой
   магазин, сотрудник — только своё
4. Объяснение на языке директора — не технические термины, а понятные
   причины и следствия
5. Минимальная стоимость — использовать уже существующие данные и
   endpoints, не изобретать новый сбор данных

## 2.2 ДАННЫЕ УЖЕ ДОСТУПНЫ ДЛЯ AI (не нужно собирать заново):

• Почасовая выручка план/факт — hourly-plan-fact возвращает кумулятивный

кассиром, суммой и составом чека
• Расписания сотрудников — /api/schedules по всем магазинам
• Мёртвые остатки — dead-stocks/update уже отправляет данные в Telegram
• KPI по сотрудникам — MVP считает score, avg_check, return_rate, margin
• Рекомендации закупки — MVP считает сезонный коэффициент, оборачиваемость

ЕСТЬ В КОДЕ (не используется):
• /api/ai/procurement-recommendations — работает, но нет UI
• /api/ai/employee-shift-kpi — работает, но нет UI
• Cron Triggers — настроены, но не обрабатывают AI-задачи
• Zod валидация — есть, но AI-ответы не валидируются

НУЖНО СОЗДАТЬ:
• Нарративный слой — Claude API для генерации текста на русском
• Push-система — триггеры для отправки алертов без запроса пользователя
• Cron jobs — ежедневные и еженедельные автоотчёты
• Диалоговый режим — tool use для ответов на свободные вопросы (Фаза 3)
• UI в Mini App — карточки инсайтов, история алертов
• KV для контекста — хранение истории диалога

// packages/backend/src/ai/client.ts
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic({ apiKey: env.ANTHROPIC_API_KEY });

export async function generateNarrative(
systemPrompt: string,
data: unknown,
maxTokens = 800
): Promise<string> {
const response = await client.messages.create({
model: env.AI_MODEL ?? "claude-sonnet-4-20250514",
max_tokens: maxTokens,
system: systemPrompt,
messages: [{
role: "user",
content: JSON.stringify(data, null, 2)
}]
});

    const text = response.content
      .filter(b => b.type === "text")
      .map(b => b.text)
      .join("");

    logger.info("AI narrative generated", {
      tokens: response.usage.output_tokens,
      feature: systemPrompt.slice(0, 50)
    });

    return text;

}

# ================================================================================ 10. РИСКИ И ПЛАН МИТИГАЦИИ

РИСК 1: Claude API недоступен или timeout
Вероятность: Средняя | Влияние: Высокое
Митигация:
• Fallback: отправить сырые данные без нарратива
• AI-функции опциональны — core функционал не затронут
• Retry с экспоненциальным backoff (1s, 2s, 4s, 8s)
• Timeout 10 секунд, не блокировать основной поток

РИСК 2: Высокая стоимость токенов при росте сети
Вероятность: Средняя | Влияние: Среднее
Митигация:
• Caching промежуточных данных в KV
• Batching нескольких магазинов в один запрос
• Prompt caching API Anthropic для повторяющихся системных промтов
• Мониторинг стоимости через Cloudflare Analytics

РИСК 3: Некачественные AI-ответы (галлюцинации)
Вероятность: Низкая | Влияние: Высокое
Митигация:
• Structured output через Zod валидацию всех AI-ответов
• Конкретные числовые данные в промте (не абстрактные описания)
• Кнопка «Неверно» в каждом Telegram-сообщении от AI
• Логирование всех ответов для ретроспективного анализа

РИСК 4: Переполнение Telegram лимитов (30 msg/sec)
Вероятность: Низкая | Влияние: Среднее
Митигация:
• Существующий queue в проекте уже решает базовую проблему
• Rate limiting для AI: не более 1 сообщения на магазин в 5 минут
• Батчинг алертов — собирать за 15 минут, отправлять дайджестом

РИСК 5: Директор игнорирует уведомления (overload)
Вероятность: Высокая | Влияние: Среднее
Митигация:
• Строгий приоритет: только критические алерты в рабочее время
• Еженедельный дайджест вместо N отдельных сообщений
• Настройки в Mini App: выбор типов уведомлений
• A/B тест частоты для оптимизации открываемости

РИСК 6: Утечка бизнес-данных через Claude API
Вероятность: Низкая | Влияние: Критическое
Митигация:
• Не передавать PII: имена клиентов, телефоны, адреса
• Передавать только агрегированные данные и UUID
• Ознакомиться с data handling policy Anthropic (zero data retention)
• Логировать что именно передаётся в каждом промте
