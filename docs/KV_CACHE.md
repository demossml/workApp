KV CACHE ARCHITECTURE
Кеширование для системы аналитики и AI-директора
Источник данных: Evotor API
Инфраструктура: Cloudflare Workers + Workers KV

====================================================
1. ПРОБЛЕМА
====================================================

Evotor API имеет ограничения:

1. медленные ответы (1–3 секунды)
2. rate limits
3. высокая нагрузка на API

Если каждый запрос фронтенда обращается к Evotor API,
система будет работать медленно.

Проблемная архитектура:

frontend
   ↓
worker
   ↓
evotor API

Каждый запрос вызывает API.

Решение — использовать кеширование через KV.

====================================================
2. АРХИТЕКТУРА КЕШИРОВАНИЯ
====================================================

Правильная схема:

Frontend
   ↓
Worker API
   ↓
KV cache
   ↓
Database
   ↓
Evotor API

Логика работы:

1. проверяем данные в KV
2. если данные есть — возвращаем
3. если нет — получаем из базы или API
4. сохраняем в KV
5. возвращаем результат

====================================================
3. СТРУКТУРА КЛЮЧЕЙ KV
====================================================

Ключи должны быть иерархическими.

----------------------------------------------------
Продажи за день
----------------------------------------------------

sales:store:{storeId}:day:{date}

пример:

sales:store:123:day:2026-03-12

----------------------------------------------------
Продажи по часам
----------------------------------------------------

sales:store:{storeId}:hour:{date}

пример:

sales:store:123:hour:2026-03-12

----------------------------------------------------
Топ товаров
----------------------------------------------------

products:top:{storeId}:{period}

пример:

products:top:123:today

----------------------------------------------------
AI отчет директора
----------------------------------------------------

ai:report:{storeId}:{date}

пример:

ai:report:123:2026-03-12

====================================================
4. TTL КЕША
====================================================

Разные данные должны храниться разное время.

Продажи сегодня
TTL = 5 минут

Продажи вчера
TTL = 24 часа

Топ товаров
TTL = 30 минут

AI отчет
TTL = 12 часов

====================================================
5. ФУНКЦИЯ КЕШИРОВАНИЯ
====================================================

Пример функции получения данных с кешем.

async function getCached(key, ttl, fetchFunction, env) {

  const cached = await env.KV.get(key)

  if (cached) {
    return JSON.parse(cached)
  }

  const data = await fetchFunction()

  await env.KV.put(
    key,
    JSON.stringify(data),
    { expirationTtl: ttl }
  )

  return data
}

6. ПРИМЕР ИСПОЛЬЗОВАНИЯ

const sales = await getCached(

====================================================
7. ИНВАЛИДАЦИЯ КЕША
====================================================

Когда появляется новый чек,
кеш нужно очищать.

событие:

receipt.created

удаляем ключи:

sales:store:{storeId}:day:{date}

sales:store:{storeId}:hour:{date}

products:top:{storeId}:today

пример:

await env.KV.delete(
  sales:store:${storeId}:day:${date}
)

====================================================
8. PREWARM КЕША
====================================================

Чтобы ускорить систему,
можно заранее обновлять кеш.

Используется cron worker.

каждые 5 минут:

1. получаем список магазинов
2. считаем продажи
3. записываем в KV

пример:

export default {

 async scheduled(event, env) {

   const stores = await getStores()

   for (const store of stores) {

     const data = await fetchSales(store.id)

     await env.KV.put(
       sales:store:${store.id}:today,
       JSON.stringify(data),
       { expirationTtl: 300 }
     )
   }

 }

}
