# Исследование: План v4.0

## Текущая архитектура

Система планов размазана по **6 точкам входа**:

| # | Компонент | Где | Строк | Что делает |
|---|-----------|-----|-------|------------|
| 1 | **PlanStatusWidget** | Home (плитки) | ~330 | Сетка карточек по магазинам: план%, прогресс-бар, план/факт ₽, статус открытия, раскрытие с товарами плана + трансфер |
| 2 | **PlanSalesFinancialReport** | `/components/` | 450 | Детальный финотчёт с мини-графиками, временем открытия, план vs факт |
| 3 | **PlanMetrics** | `/pages/reports/` | 398 | Полная страница с метриками, целями, прогрессом по магазинам |
| 4 | **PlanMetricsCard** | `/components/` | 290 | Карточка-версия PlanMetrics |
| 5 | **DailyBriefing** | Home | ~167 | Утренняя сводка — использует planData |
| 6 | **Topbar** | Шапка | ~30 | Показывает статус плана в топбаре |

### Поток данных

```
useGetReportAndPlan (React Query, refetch каждые 60с)
  └─ fetchReportAndPlanForToday()
       ├─ fetchFinancialForToday() → reportData (продажи, возвраты, чеки)
       └─ fetchPlanForToday() → planData { shopName: { datePlan, dataSales, dataQuantity } }
            └─ Backend: /api/evotor/plan-for-today
                 ├─ DuckDB plan table (daily_plan по shop_uuid)
                 └─ vape_groups JOIN positions → dataSales (только вейпы!)
```

## Найденные проблемы

### 1. Дублирование логики
Одни и те же вычисления в 3+ местах:
- `calculatePlanProgress` — свой в PlanStatusWidget (через planStatusModel), свой в PlanMetrics, свой в PlanSalesFinancialReport
- `formatAmount` — везде своя реализация
- Статусы (green/yellow/red) — PlanStatusWidget использует 70% порог, PlanMetrics — 70%/90%

### 2. dataSales ≠ общие продажи
Backend `plan-for-today` считает **только вейп-продажи** (JOIN vape_groups).
Но PlanMetrics на фронте считает `totalSell - refund` из `reportData` — другие цифры!
PlanStatusWidget берёт `dataSales` из planData (вейпы) — показывает не полную выручку магазина.

### 3. Три разных UI для одних данных
Пользователь видит план в:
- Плитках на Home (PlanStatusWidget) — красиво, но мало деталей
- Странице PlanMetrics — детально, но отдельная страница
- PlanSalesFinancialReport — ещё один вариант, сейчас не используется на Home

Нет единого, нарастающего по детализации представления.

### 4. Нет временного измерения
Всё — «сегодня». Нельзя посмотреть:
- Как шёл план по часам (есть в SalesTempoWidget, но отдельно)
- План вчера / неделю назад
- Динамику выполнения за последние 7 дней

### 5. Трансфер-рекомендации не на месте
Кнопка «Рекомендации по перемещению» в PlanStatusWidget делает отдельный API-запрос. Это отдельная фича, смешанная с планом.

### 6. Статус открытия смешан с планом
Время открытия и имя продавца — полезно, но это другая сущность. Сейчас вшито в PlanStatusWidget.

### 7. Нет консолидированной сводки
PlanStatusWidget показывает only per-shop cards. Нет общей строки «Всего по сети: план X / факт Y / Z%».

## Гипотезы для v4.0

### H1: Единая модель данных плана
Вынести ВСЮ логику расчёта плана в `planStatusModel` (или новый `planModel`).
Все виджеты используют одну модель. Статусы, форматирование, пороги — в одном месте.

### H2: План = общая выручка (не только вейпы)
Изменить backend так, чтобы `dataSales` в planData был равен общей выручке магазина (`totalSell - refund`), а не только вейпам.
Вейпы оставить как подметрику.

### H3: Консолидированная сводка по сети
Добавить над сеткой карточек строку: «Сеть: план X / факт Y / Z% • N из M магазинов выполняют план».

### H4: Объединить PlanStatusWidget + SalesTempoWidget
SalesTempoWidget уже показывает план-факт по часам. Если объединить с PlanStatusWidget:
- Свёрнуто: сетка карточек как сейчас
- Раскрыто (тап на магазин): почасовая динамика + товары плана + прогноз

### H5: Вкладки «Сегодня / Вчера / Неделя»
DateFilter уже есть на Home. План должен реагировать на выбранный период — показывать план на выбранную дату.

### H6: Вынести статус открытия в отдельный виджет
Рядом с планом, но отдельно — «Статус магазинов»: кто открыт, кто опаздывает, время открытия.

### H7: Убрать трансфер-рекомендации из PlanStatusWidget
Либо в отдельный виджет, либо в отдельную вкладку/страницу.

## Приоритеты (на обсуждение)

1. **H1** — фундамент, без него остальное бессмысленно
2. **H3** — максимальный UX-эффект при минимальных затратах
3. **H2** — правильные данные (сейчас план показывает вейпы, а не выручку)
4. **H6** — разделение ответственности
5. **H4 / H5** — более глубокая переработка
